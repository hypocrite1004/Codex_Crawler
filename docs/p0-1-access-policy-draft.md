# P0-1 권한 정책표 초안

기준일: 2026-03-27  
상태: 확정

## 승인 반영 현황

### 확정된 항목
- `published` 게시글 상세는 비로그인 사용자에게 공개합니다.
- 운영 대시보드는 `staff/admin` 전용으로 둡니다.
- `staff`와 `admin`은 실제 코드에서도 분리합니다.
- 댓글 전역 API는 유지보다 축소합니다.
- 게시글 삭제 권한은 `admin`만 가집니다.
- 크롤러 `preview/manual crawl`은 `admin`만 수행합니다.

## 1. 정책 원칙

### 원칙 1. 공개 콘텐츠와 운영 데이터는 분리합니다
- `published` 게시글과 공개 CVE 데이터만 public 영역으로 둡니다.
- 대시보드, 크롤러, AI 설정, 운영 통계는 기본적으로 운영 영역으로 둡니다.

### 원칙 2. 역할과 소유권을 분리합니다
- `user`, `staff`, `admin`은 계정 역할입니다.
- `author`는 별도 계정 역할이 아니라 `리소스 소유자`를 뜻하는 논리 역할입니다.
- 즉, `author`는 `자신이 작성한 게시글/댓글/요약`에 대해서만 추가 권한을 가집니다.

### 원칙 3. `staff`와 `admin`은 분리하는 방향으로 갑니다
- 현재 코드상 주요 운영 권한은 대부분 Django `is_staff` 하나에 묶여 있습니다.
- 초안 정책에서는 아래처럼 나눕니다.
  - `staff`: 콘텐츠 운영 및 검수 담당
  - `admin`: 시스템 운영 및 민감 설정 담당
- 이 분리가 확정되기 전까지는 현재 코드와 정책 사이에 차이가 남습니다.

### 원칙 4. 읽기 권한은 넓게, 쓰기 권한은 좁게 둡니다
- 공개 정보는 읽기 쉽게 유지합니다.
- 변경 권한은 `소유자`, `staff`, `admin` 중심으로 제한합니다.

### 원칙 5. 운영 리스크가 큰 기능은 `admin` 전용으로 둡니다
- 크롤러 source 수정
- 수동 crawl 실행
- AI 설정 수정
- 민감 토큰/헤더/요청 바디 열람

## 2. 역할 정의

| 역할 | 정의 |
|---|---|
| `guest` | 비로그인 사용자 |
| `user` | 로그인한 일반 사용자 |
| `author` | 특정 게시글/댓글의 작성자 |
| `staff` | 콘텐츠 검수/운영 담당자 |
| `admin` | 시스템/보안/운영 설정 담당자 |

주의:
- 현재 코드에는 `staff`와 `admin`을 구분하는 별도 권한 계층이 없습니다.
- 확정 정책상 `admin`은 별도 권한 계층으로 분리 대상이며, 현재는 대부분 `staff`가 대신 수행하고 있습니다.

## 4. 현재 코드와 충돌하는 지점

### 4-1. `staff`와 `admin`이 아직 실제 코드에서 분리되어 있지 않습니다
- 현재 운영성 엔드포인트 대부분은 Django `IsAdminUser` 기준이라 `is_staff`와 사실상 동일합니다.
- 확정 정책의 `admin 전용` 기능이 현재는 `staff`에게 열려 있습니다.

### 4-2. 대시보드가 일반 로그인 사용자에게 열려 있습니다
- 현재는 인증만 되면 운영 대시보드 접근이 가능합니다.
- 초안 정책은 `staff/admin` 전용을 권장합니다.

### 4-3. 공유 상태 변경이 소유권과 무관하게 열려 있습니다
- 현재 `toggle_share`는 인증만 요구합니다.
- 초안 정책은 `author/staff/admin`만 허용합니다.

### 4-4. 댓글 전역 API가 공개 범위보다 넓습니다
- 현재 댓글은 전역 queryset 기반으로 직접 조회 가능합니다.
- 초안 정책은 게시글 공개 정책과 연동된 제한을 권장합니다.

### 4-5. 게시글 상세 공개 정책이 프론트와 백엔드에서 어긋납니다
- 백엔드는 `published` 게시글을 공개 대상으로 볼 수 있는 구조입니다.
- 프론트 상세 페이지는 로그인 가드가 걸려 있습니다.

### 4-6. 크롤러/AI 설정의 민감 기능이 운영자 계층 전체에 묶여 있습니다
- 정책상 `admin 전용`으로 두는 것이 맞는 기능이 현재 `staff`에 열려 있습니다.
