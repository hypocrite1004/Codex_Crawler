# P1-10 핵심 QA 시나리오 정의

## 목적

현재 프로젝트는 권한, 공개/비공개 정책, 크롤러 운영, CVE 흐름이 서로 연결돼 있다.  
따라서 QA도 페이지 단위가 아니라 `핵심 사용자 흐름` 기준으로 관리해야 한다.

이 문서는 최소 수동 QA 기준선을 정의한다.

## 공통 계정

- `guest`
- `author`
- `staff`
- `admin`

가능하면 검증용 계정은 아래 역할로 고정한다.

- `qa_author`
- `qa_staff`
- `qa_admin`

## 시나리오 1. 비로그인 홈 피드 → 게시글 상세

### 목적

- 공개 게시글 접근 정책 검증
- 비공개 게시글 보호 검증

### 절차

1. `guest` 상태에서 홈 피드 진입
2. 공개 게시글 상세 진입
3. 비공개 게시글 URL 직접 접근 시도

### 기대 결과

- 공개 게시글 상세는 접근 가능
- 비공개 게시글은 `404` 또는 동등한 수준으로 숨김
- 공개 게시글 상세에서 댓글 목록은 보이되, 댓글 작성은 로그인 유도 문구 노출

## 시나리오 2. 작성자 초안 저장 → 검토 요청

### 목적

- 작성자 워크플로 검증

### 절차

1. `author` 로그인
2. 게시글 작성 화면 진입
3. 초안 저장
4. 내 프로필에서 초안 확인
5. 검토 요청 실행

### 기대 결과

- 초안 상태로 저장 가능
- 프로필에서 상태가 `draft`로 보임
- 검토 요청 후 상태가 `review`로 변경

## 시나리오 3. 관리자 승인/반려 → 초안 복원

### 목적

- staff/admin 운영 워크플로 검증

### 절차

1. `staff` 또는 `admin` 로그인
2. `admin/posts`에서 review 상태 게시글 확인
3. 승인 또는 반려 실행
4. 작성자 계정에서 상태 확인
5. 필요 시 `restore to draft` 실행

### 기대 결과

- 운영자는 review 게시글 승인/반려 가능
- 작성자는 상태 변화 확인 가능
- review/archived 상태 게시글은 draft로 복원 가능

## 시나리오 4. 크롤러 preview → crawl → logs/runs/items

### 목적

- admin 전용 크롤러 운영 흐름 검증

### 절차

1. `admin` 로그인
2. `admin/crawler` 진입
3. source 생성 또는 기존 source 선택
4. preview 실행
5. manual crawl 실행
6. logs 확인
7. runs 확인
8. run items 확인

### 기대 결과

- `staff`는 진입 또는 API 호출 불가
- `admin`은 preview/manual crawl 가능
- logs/runs/items 데이터가 연결되어 조회 가능
- private IP / 금지 헤더 / 잘못된 URL은 차단됨

## 시나리오 5. CVE 목록 → 상세 → 관련 게시글 → 상세 페이지

### 목적

- CVE 공개 흐름 검증

### 절차

1. `guest` 상태에서 `/cves` 진입
2. CVE 상세 진입
3. 관련 게시글 목록 확인
4. 연결된 게시글 상세 진입

### 기대 결과

- 공개 CVE 목록/상세 접근 가능
- 운영 필드(`notes`, `is_tracked`, `legacy_mention_count`)는 비노출
- 관련 게시글 링크가 정상 동작

## 시나리오 6. 운영자 UI 권한 노출

### 목적

- staff/admin UI 권한 노출 정합성 검증

### 절차

1. `staff` 로그인 후 게시글 상세 진입
2. 요약 영역 확인
3. 댓글 영역 확인
4. `admin` 로그인 후 동일 흐름 확인

### 기대 결과

- `staff`는 요약 생성/편집/삭제 버튼 확인 가능
- `staff`는 타 사용자 댓글에 대해 편집/삭제 버튼 확인 가능
- `admin`은 `admin/crawler`, `admin/ai` 접근 가능
- `staff`는 `admin/posts` 접근 가능하지만 `admin/crawler`, `admin/ai`는 접근 불가

## 시나리오 7. 인증/세션 흐름

### 목적

- 토큰 저장/세션 갱신 흐름 검증

### 절차

1. 로그인
2. 보호 페이지 진입
3. 새로고침
4. 세션 만료 후 API 요청

### 기대 결과

- 로그인 후 세션이 저장됨
- 보호 페이지 접근 가능
- 만료 시 refresh 재시도 후 복구 또는 로그아웃 처리
- `NEXT_PUBLIC_API_URL` 미설정 시 로컬 fallback 동작

## 최소 완료 기준

- 위 7개 시나리오를 기준으로 수동 QA 가능해야 함
- 새 기능 추가 시 이 시나리오 중 어느 흐름에 영향이 있는지 먼저 표시해야 함
